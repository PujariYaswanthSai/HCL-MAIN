Flask App Factory Pattern

The app factory pattern keeps Flask clean and testable. All extensions are initialized once and imported by blueprints as needed.

# app/__init__.py from flask import Flask from .extensions import db, jwt, migrate, cors from .blueprints.auth_bp import auth_bp from .blueprints.vehicles_bp import vehicles_bp from .blueprints.bookings_bp import bookings_bp from .blueprints.payments_bp import payments_bp from .blueprints.maintenance_bp import maintenance_bp def create_app(config_name='development'): app = Flask(__name__) app.config.from_object(config[config_name]) # Init extensions db.init_app(app) jwt.init_app(app) migrate.init_app(app, db) cors.init_app(app, resources={r"/api/*": {"origins": "*"}}) # Register blueprints with /api prefix app.register_blueprint(auth_bp, url_prefix='/api/auth') app.register_blueprint(vehicles_bp, url_prefix='/api/vehicles') app.register_blueprint(bookings_bp, url_prefix='/api/bookings') app.register_blueprint(payments_bp, url_prefix='/api/payments') app.register_blueprint(maintenance_bp, url_prefix='/api/maintenance') return app

Core Business Logic — Booking Service

# services/booking_service.py from app.models.booking import Booking from app.services.pricing_engine import calculate_price from app.utils.responses import error_response def check_availability(vehicle_id, pickup_time, return_time): """Returns True if no overlap with confirmed bookings.""" conflict = Booking.query.filter( Booking.vehicle_id == vehicle_id, Booking.status.in_(['confirmed', 'active']), Booking.pickup_time < return_time, # existing starts before new ends Booking.return_time > pickup_time # existing ends after new starts ).first() return conflict is None def create_booking(user_id, vehicle_id, pickup_time, return_time, location): if not check_availability(vehicle_id, pickup_time, return_time): return None, "Vehicle not available for selected dates" total_price = calculate_price(vehicle_id, pickup_time, return_time) booking = Booking( user_id=user_id, vehicle_id=vehicle_id, pickup_time=pickup_time, return_time=return_time, pickup_location=location, total_price=total_price, status='pending' ) db.session.add(booking) db.session.commit() return booking, None

RBAC — Role Decorator

# utils/decorators.py from functools import wraps from flask_jwt_extended import get_jwt from app.utils.responses import error_response def role_required(*roles): def decorator(fn): @wraps(fn) def wrapper(*args, **kwargs): claims = get_jwt() if claims.get('role') not in roles: return error_response("Insufficient permissions", 403) return fn(*args, **kwargs) return wrapper return decorator # Usage on any route: @vehicles_bp.route('/', methods=['POST']) @jwt_required() @role_required('admin') def add_vehicle(): ...

JWT Auth Flow — Step by Step

01
POST /api/auth/login with credentials
React sends {email, password} as JSON body. No CSRF needed — JWT is stateless.
// React axiosInstance.js
axios.post('/api/auth/login', { email, password })
02
Flask verifies password with bcrypt, signs tokens
access_token expires in 15 min. refresh_token expires in 7 days. Role claim embedded in JWT payload.
create_access_token(identity=user.id, additional_claims={'role': user.role})
create_refresh_token(identity=user.id)
03
React stores tokens and attaches on every request
access_token stored in memory (AuthContext). refresh_token in httpOnly cookie. Axios request interceptor auto-attaches the Bearer header.
// Request interceptor
config.headers.Authorization = `Bearer ${getAccessToken()}`
04
Flask decodes JWT and checks role on every protected route
@jwt_required() decorator decodes token. @role_required('admin') checks the role claim. 401 if invalid token, 403 if wrong role.
05
Axios response interceptor silently refreshes expired tokens
On 401 response, interceptor calls /api/auth/refresh, gets new access_token, updates AuthContext, retries the original failed request transparently.
Role 04 — Database Engineer

Database

SQLite via SQLAlchemy ORM. 9 tables normalized to 3NF. All relationships enforced with foreign keys. Migrations managed with Flask-Migrate (Alembic). Designed to be dropped-in replaceable with PostgreSQL in production with zero model changes.

DB

Database Setup & Configuration

Engine
SQLite (file-based). Located at instance/rental.db. Zero setup required for development.
ORM
SQLAlchemy — model-first. All queries go through the ORM. No raw SQL except for complex aggregate reports.
Migrations
Flask-Migrate wraps Alembic. Run flask db migrate + flask db upgrade to apply.
Production swap
Change DATABASE_URL in .env to a PostgreSQL URI. No model code changes required — SQLAlchemy abstracts the dialect.
Normalization
All 9 tables are in 3NF. PricingRules is a separate table — not embedded in Vehicles. StatusLog appends, never overwrites.
Seeding
A seed.py script populates sample vehicles, users (one per role), categories, and pricing rules for the demo.
# extensions.py — shared db instance from flask_sqlalchemy import SQLAlchemy from flask_migrate import Migrate db = SQLAlchemy() migrate = Migrate() # config.py class DevelopmentConfig: SQLALCHEMY_DATABASE_URI = 'sqlite:///rental.db' SQLALCHEMY_TRACK_MODIFICATIONS = False # commands to initialize: # flask db init → creates migrations/ folder # flask db migrate → auto-generates migration file from models # flask db upgrade → applies migration to SQLite file

Table Relationships

users→bookings1 — NOne user can have many bookings. Booking.user_id → users.id
vehicles→bookings1 — NOne vehicle can have many bookings (across different time windows). Booking.vehicle_id → vehicles.id
bookings→payments1 — 1Each booking has exactly one payment record. Payment.booking_id is UNIQUE.
bookings→booking_status_log1 — NEvery status transition creates a new log row. Append-only — never updated.
bookings→reviews1 — 1One review per completed booking. Review.booking_id is UNIQUE.
vehicles→maintenance1 — NOne vehicle can have many maintenance records over its lifetime.
vehicle_categories→vehicles1 — NOne category (e.g. "SUV") groups many vehicles. Pricing is category-based.
vehicle_categories→pricing_rules1 — NPricing rules can be category-wide or platform-wide (category_id nullable).
